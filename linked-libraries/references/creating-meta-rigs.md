## Creating Meta-rigs

Creating meta-rigs. Outline: add a single bone from Add ▸ Armature, enter armature Edit Mode and assemble it from samples or Rigify types, then set up the Rigify bone-collection UI along with color sets and selection sets, and click Generate in the armature properties.

How Rigify works. A meta-rig is split into multiple sub-rigs: it's an assembly of bone chains, where a chain is identified by the Connected attribute, and chains can be joined further by parenting them without Connected (the Keep Offset option). Each chain's first bone carries a custom attribute — a Rigify property naming the sub-rig type — and at generation Rigify walks each chain first-bone-to-last to decide which controls and deform bones to build. Because a meta-rig is just a collection of sub-rigs, new ones are assembled by combining sub-rig samples in different ways, yielding effectively unlimited meta-rigs from the same blocks. The meta-rig holds more than the visible bones: at generation Rigify identifies each sub-rig type and, per the chosen options, builds all the controls, switches, and deform bones in one click.

Creating a new meta-rig. Add a new armature object (Object Mode; Add ▸ Armature ▸ Single Bone; Shift-A) — you need an armature to work with, so add a single bone; naming it "metarig" early is a good idea so it's always clear which armature you're editing. Then, with it selected, enter Edit Mode and build either by adding samples or by creating bone chains. Adding samples (basic) is a good way to learn the blocks: go to the armature tab, scroll to the Rigify panel, pick a sample, click Add sample, and edit bone positions to fit (see the Rig Types page for the list). Using Rig Types (advanced): for full control, use the Rigify Type panel in bone properties (Pose Mode) to assign any sub-rig type to any bone and set its options — the panel's top field sets the active bone's rig type (optionally filtered by Feature Set), with relevant options below. Some control-heavy rig types reference Bone Collections through lists that share a standard UI: a checkbox to use the reference, a button to copy the list from active to selected bones, a plus to add a reference, and per-entry a target-collection field with a remove button. Each sub-rig needs a required number of input bones, so if unsure, add a sample to see how it's meant to be used. Preserved bone properties: certain metarig-bone properties are often copied to generated control, deform, and mechanism bones, the exact set depending on the sub-rig and bone — common groups are Parenting Settings (parent ORG bone, Use Connect, Use Inherit Rotation, Use Local Location, Inherit Scale; usually copied to deform bones and FK controls), Bendy Bone Settings in Edit Mode (segment count, Vertex Mapping Mode, Ease/Roll/Curve/Scale In/Out; segment count often overridden by a sub-rig option, the rest usually copied to deform bones), Transformation Settings (rotation mode, pose rotation values, channel locks; usually copied to FK controls), Custom Properties (usually copied to one generated control, mainly FK, with intra-armature drivers retargeted to the copy), and Custom Widget (copied to one generated control, mainly FK, suppressing automatic widget generation). Custom root bone: if the meta-rig has a bone named root, it becomes the root control instead of a new one, letting you change the root's rest position, give it a custom widget, or add custom properties.

Such a root bone needs no parent, and its sub-rig type should be basic.raw_copy (or left unset). Color Sets: this panel defines the final rig's bone color scheme, and its colors can be tied to bone collections from the relevant panel. The panel's top two rows govern overall color behavior — color themes usually run a gradient across default, selected, and active states, which makes spotting the selected or active bone hard when several themes coexist, so Rigify unifies the selected and active states to one color via two settings: Unified Selected/Active Colors (when on, adding a bone group keeps colors consistent; a scheme added from a theme loads as-is, and the Apply button forces unification) and the Selected/Active Colors fields (the two colors, read by default from the user's Blender preferences theme so the UI stays predictable, customizable per field and resettable to the current theme via the update-icon button). Color sets are added and removed with the +/- buttons, cleared all at once from the Specials menu, populated with Rigify's default scheme via Add Standard, or filled from a chosen theme via Add From Theme. Bone Collections UI: bone collections group related bones so they hide/reveal together, and Rigify uses them to generate extra features plus a Rig Layers panel of buttons for hiding collections in an intuitive layout. This panel configures that generated layout and other collection settings (like the color set). Its top is a list mirroring the main bone-collection list but with extra columns — color set, whether the collection has a button, whether it makes a selection set. Validate Collection References runs a scan that validates and normalizes all collection references (Rigify's referencing survives renames, but deleting collections or joining armatures can break references), reporting errors and reducing future breakage; it also runs automatically at every generation, and to stay safe it should be run both right before and right after joining two metarig armatures — specifically, always between renaming collections and joining. Per-collection settings: Color Set picks the collection's color (when a bone is in several collections, the earlier one in the list usually wins); Add Selection Set toggles generating a selection set; UI Row, if nonzero, places the collection's visibility button on that Rig Layers row (zero means no button and the collection stays hidden); UI Title overrides the button's label (titles needn't be unique, so they can cut clutter using context). The UI Layout sub-panel is a WYSIWYG editor for that generated panel (driven by UI Row/UI Title): each row ends with three buttons — Arrow (move the active collection's button here), Plus (insert a row before this one), Minus (remove the row and shift buttons up) — while to their left the rows show collection buttons (clicking selects the collection rather than hiding it), and the active collection's button becomes a UI Title field plus an X to unassign it; unassigned collections appear in a separate section at the bottom. The Root collection gets a UI button automatically at generation if needed, and you may manually assign buttons to the internal ORG, DEF, and MCH collections; blank rows render much thinner in the final UI and work as separators. Actions: the Action constraint applies poses from an action to bones based on another bone's transform, which normally means adding the constraint to every affected bone — tedious — so Rigify's Actions panel automates it. The panel lists actions to apply to generated bones (each action listed once); each entry shows its action name, what triggers it (either a bone or a corrective action that two others drive), and a disable checkbox, with the leading icon switching from an action glyph to a link glyph to flag corrective actions that depend on the active normal action — or the normal actions a selected corrective action relies on.

Rigify adds the Action constraints in an order that reproduces the intended deformation, assuming the actions were posed and keyframed in the listed order. Normal actions are driven by a specific generated control bone and expose: Control Bone (the driving bone); Symmetrical (when the control's name carries a left/right suffix, auto-applies symmetry, so left-side keyframed bones follow the left control and right-side bones the right, while side-less bones are treated as centered and get two Action constraints at influence 0.5, one per control); Frame Start & End (the action's frame range used by the created constraints); Target Space and Transform Channel (the target bone's coordinate space and channel); Min/Max (the range of channel values mapped onto the action's frame range); and Default Frame (the action frame mapping to the channel's neutral value — 1 for scale, 0 otherwise — computed from the range). Corrective actions are driven by the progress of two other listed actions and refine the pose when both are used together: they expose Frame Start & End, and Trigger A & B (the two controlling actions, with row buttons to show their settings or jump to them). A corrective action's progress from start to end is the product of its two triggers' progress, so its start frame applies when either trigger is at its start and its end frame when both are at their end; corrective actions must sit below their triggers in the list (enforced by an implicit reorder if violated). They behave most intuitively when both triggers have Default Frame equal to Start Frame — to make one in that case, create the two triggers, add them to the panel, and generate; pose your controls so both triggers reach their end frame; pose and keyframe the corrections on the new action's end frame while keying the start frame to neutral; then add the new action to the end of the list and configure it.

## Application Templates

Application templates let you define a reusable configuration that can replace the default one without a separate Blender install or overwriting your personal settings. Pick a template from the splash screen or the File ▸ New submenu (the menu is hidden on the splash when no templates are found); install new ones from the Blender menu, and save your preferences if you want the active template to persist across restarts. Motivation: sometimes a single script or add-on isn't enough and you don't want to ask someone to swap their preferences and startup file, install scripts, and change their keymap — templates let people switch to a customized configuration without disrupting their own settings or install, so they can build and easily distribute their own applications on top of Blender. Details: a template can supply its own startup file — the one loaded by default; preferences (only a subset is used: themes, add-ons, keymaps, and viewport lighting); splash-screen image; and Python scripts (with the same capabilities as any script, typically modifying or replacing parts of the UI, defining new menus/keymaps/tools, or pointing to a custom add-on path holding add-ons specific to the template). A template also keeps its own user configuration, so saving a startup file while using one won't overwrite your default. Directory layout: templates live in one of two spots under the scripts directory — {`BLENDER_USER_SCRIPTS`}/startup/bl_app_templates_user or {`BLENDER_SYSTEM_SCRIPTS`}/startup/bl_app_templates_system — and user configuration is stored in a subdirectory: without a template at ./config/startup.blend and ./config/userpref.blend, and with one at ./config/{`APP_TEMPLATE_ID`}/startup.blend and ./config/{`APP_TEMPLATE_ID`}/userpref.blend (see Blender's Directory Layout for script and config locations). If paths aren't being found, log Blender's path look-ups, e.g. `blender --log "bke.appdir.*" --log-level -1 --app-template my_app_template`, then check where it tries to access the template. You can also set up a launcher that opens Blender with a given template via `blender --app-template my_template`. Template contents (each file optional): startup.blend (the factory startup file), userpref.blend (the factory preferences file; when omitted, preferences are shared with the default config, and only a subset is used), splash.png (a splash override, which must be 1000x500, excluding header text), and __init__.py (a Python script that must contain register and unregister functions). The bundled startup.blend and userpref.blend count as Factory Settings and are never overwritten; the user can save their own startup/preferences while using the template (stored in their user config) only when the template includes its own userpref.blend, and the original template settings reload via Load Template Factory Settings in the file menu, much like Load Factory Settings. Template scripts: app templates use the same APIs as add-ons and other scripts; an optional __init__.py lets you change the startup or preferences without shipping a blend-file and lets you make changes dynamically (for example, checking the processor count, OS, and memory and setting values accordingly), and lets you enable add-ons tied to your template. The register function runs on activation and unregister when another template is selected; since these run once, any changes to defaults must go through a handler — the two you'll likely use are bpy.app.handlers.load_factory_preferences_post and bpy.app.handlers.load_factory_startup_post — which let you define your own "factory settings" the user can change, just as Blender has its own defaults on first launch. The page includes an example __init__.py defining such defaults:
```text
import bpy
from bpy.app.handlers import persistent

@persistent
def load_handler_for_preferences(_):
    print("Changing Preference Defaults!")
    from bpy import context
    prefs = context.preferences
    prefs.use_preferences_save = False
    kc = context.window_manager.keyconfigs["blender"]
    kc_prefs = kc.preferences
    if kc_prefs is not None:
        kc_prefs.select_mouse = 'RIGHT'
        kc_prefs.spacebar_action = 'SEARCH'
        kc_prefs.use_pie_click_drag = True
    view = prefs.view
    view.header_align = 'BOTTOM'

@persistent
def load_handler_for_startup(_):
    print("Changing Startup Defaults!")
    for mesh in bpy.data.meshes:
        for poly in mesh.polygons:
            poly.use_smooth = True
    for screen in bpy.data.screens:
        for area in screen.areas:
            for space in area.spaces:
                if space.type == 'VIEW_3D':
                    space.shading.type = 'MATERIAL'
                    space.shading.use_scene_lights = True

def register():
    bpy.app.handlers.load_factory_preferences_post.append(load_handler_for_preferences)
    bpy.app.handlers.load_factory_startup_post.append(load_handler_for_startup)

def unregister():
    bpy.app.handlers.load_factory_preferences_post.remove(load_handler_for_preferences)
    bpy.app.handlers.load_factory_startup_post.remove(load_handler_for_startup)
```
