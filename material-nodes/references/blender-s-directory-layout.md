## Blender’s Directory Layout

This page documents the directories Blender uses, which helps with troubleshooting, automation, and customization. User directories hold preferences, the startup file, installed extensions, presets, and more, defaulting to each OS's standard config folders: on Linux, $HOME/.config/blender/5.1/ (or $`XDG_CONFIG_HOME`/blender/5.1/ when $`XDG_CONFIG_HOME` is set); on macOS, /Users/$USER/Library/Application Support/Blender/5.1/; on Windows, %USERPROFILE%\AppData\Roaming\Blender Foundation\Blender\5.1\. For a portable install (running from a removable drive), create a folder named "portable" — on Windows and Linux next to the Blender executable in the unzipped folder, and on macOS inside the bundle at Blender.app/Contents/Resources — and it will then hold preferences, the startup file, installed extensions, and presets. The `BLENDER_USER_RESOURCES` environment variable can point the user directory elsewhere. System directories hold the files bundled with Blender that it needs to run (scripts, presets, essential assets, and more): on Linux, ./5.1/ for the blender.org archive or /usr/share/blender/5.1/ for distribution packages; on macOS, ./Blender.app/Contents/Resources/5.1/; on Windows, ./5.1/ for the blender.org zip, %ProgramFiles%\Blender Foundation\Blender\5.1\ for the installer, or %ProgramFiles%\WindowsApps\BlenderFoundation.Blender\Blender\5.1\ for the Microsoft Store install. `BLENDER_SYSTEM_SCRIPTS` and `BLENDER_SYSTEM_EXTENSIONS` can add extra scripts and extensions outside the regular install, and other `BLENDER_SYSTEM` variables can override system paths (rarely used). Path layout: ./autosave holds autosave blend-files (Windows only; other systems use the temp directory) in the user directories; ./config holds user configuration and session info; ./config/startup.blend is the file loaded at startup, ./config/userpref.blend the preferences, ./config/bookmarks.txt the File Browser bookmarks, ./config/recent-files.txt the recent-file list, and ./config/{`APP_TEMPLATE_ID`}/startup.blend and userpref.blend the per-template startup and preferences; ./datafiles holds runtime data files in both user and system directories (user data files override or add to system ones), including ./datafiles/colormanagement (the default OpenColorIO config), ./datafiles/fonts (UI fonts), and ./datafiles/studiolights (3D-viewport studio lights); ./extensions holds extension repositories in both locations; and ./scripts holds add-ons, presets, templates, the UI, and startup scripts in both locations — within it, ./scripts/addons/*.py are Preferences-enabled add-ons (import/export support, render-engine integration, utilities), ./scripts/addons/modules/*.py are modules add-ons can import (added to sys.path), ./scripts/addons_core/*.py are the bundled add-ons with their own ./scripts/addons_core/modules/*.py, ./scripts/modules/*.py are the core API and utility modules (added to sys.path), ./scripts/startup/*.py are auto-imported at startup, ./scripts/startup/bl_app_templates_user/{`APP_TEMPLATE_ID`} and bl_app_templates_system/{`APP_TEMPLATE_ID`} are the user and system application templates, ./scripts/presets/{preset}/*.py store user-defined settings (cloth, render formats, etc.), ./scripts/templates_py/*.py are example scripts under Text Editor ▸ Templates ▸ Python, ./scripts/templates_osl/*.osl are example OSL shaders under Text Editor ▸ Templates ▸ Open Shading Language, and ./python is the bundled Python distribution (system directories). The local cache directory stores persistent caches (currently only Asset Library indexing) and the OS won't clear it automatically: Linux uses $`XDG_CACHE_HOME`/blender/ (or $HOME/.cache/blender/), macOS /Library/Caches/Blender/, and Windows %USERPROFILE%\AppData\Local\Blender Foundation\Blender\Cache\. The temporary directory stores various run-time files (render layers, physics cache, copy-paste buffer, crash logs), chosen by priority: the File Paths user preference, then environment variables (TEMP on Windows, TMP and `TMP_DIR` elsewhere), then /tmp/.

## Rendering From The Command Line

You might render from the command line to speed up rendering, to drive Blender remotely, or to script it — and one benefit is that no graphical display is needed (no X server on Linux), so you can render over a remote shell such as SSH. See Command Line Arguments for the full list (to choose the scene, end frame, and so on), or run `blender --help`, and see Command Line Launching for platform specifics. Remember that arguments execute in order, so `blender -b file.blend -a -x 1 -o //render` fails (the output and extension are set after the render is requested) while `blender -b file.blend -x 1 -o //render -a` works — always put -f or -a last. Single image: `blender -b file.blend -f 10`, where -b renders without the UI, file.blend is the file, and -f 10 renders only frame 10. In `blender -b file.blend -o /project/renders/frame_##### -F OPEN_EXR -f -2`, -o sets the save path with five zero-padded frame digits, -F `OPEN_EXR` overrides the blend-file's format to OpenEXR, and -f -2 renders the second-to-last frame; note arguments are case sensitive, so -F and -f differ. Animation: `blender -b file.blend -a` renders the whole animation using the blend-file's saved settings, while `blender -b file.blend -E CYCLES -s 10 -e 500 -t 2 -a` uses the Cycles render engine (run `blender -E help` for the list), sets the start frame to 10 and end to 500, and uses two threads. Cycles also has extra options beyond those shared by all engines — see Cycles Render Options.

## Deploying Blender in Production

Deploying Blender in production (studios, schools): such environments often have particular needs around security, automated deployment, and customization. Installing: a Blender download extracts to any directory as a self-contained install, several versions can coexist on one machine, and rollout can be automated with ordinary file tools. Because new versions may add, drop, or change behavior that affects production files, it's wise to pin one LTS version per project (LTS releases get two years of bug fixes). Working offline: a workstation may lack internet for security or other reasons. Blender doesn't reach the internet by default, but the System preferences' Online Access option enables it; you can enforce offline operation with the --offline-mode argument, after which users can't turn online access back on in the preferences. (Add-ons that respect this setting only connect when enabled, but Blender can't stop a third-party add-on from breaking the rule.) Bundling extensions: to hand the same extensions to everyone offline or in a controlled setup, use the default read-only System repository, which can live on a read-only network drive or a system directory. The $`BLENDER_SYSTEM_EXTENSIONS` environment variable sets its default location — point it at a directory that contains a "system" subdirectory — and extract extension packages into that system directory, giving a path like `$BLENDER_SYSTEM_EXTENSIONS/system/my-addon/blender_manifest.toml`; in the Extensions preferences you can also set a custom directory for the default System repository or create several repositories. Bundling scripts: beyond extensions, you can bundle scripts for presets, application templates, legacy add-ons, and startup. Add script directories in the File Paths preferences, or use $`BLENDER_SYSTEM_SCRIPTS` to add them without touching the preferences; a script directory is expected to hold specific subdirectories (presets, addons, startup, etc.) per the Path Layout. Startup scripts: the Python API can customize Blender — adjusting preferences, the startup file, and UI — for instance enabling an add-on for everyone via `$BLENDER_SYSTEM_SCRIPTS/startup/enable_addons.py`:
```text
def register():
    import addon_utils
    addon_utils.enable("my-addon")

def unregister():
    pass

if __name__ == "__main__":
    register()
```
Application templates set Blender up for a task or project apart from the default config (chosen when creating a file), placed like `$BLENDER_SYSTEM_SCRIPTS/startup/bl_app_templates_system/MyTemplate/__init__.py` and `.../MyTemplate/startup.blend`. Legacy add-ons not yet converted to extensions go in the addons script directory, e.g. `$BLENDER_SYSTEM_SCRIPTS/addons/simple_addon.py` or `.../addons/complex_addon/__init__.py`. Splash screen: to show which build artists are running, `BLENDER_CUSTOM_SPLASH` swaps out the entire splash image, whereas `BLENDER_CUSTOM_SPLASH_BANNER` merely lays a banner over it. VFX Platform: Blender follows the VFX reference platform, so it runs alongside other VFX tools and can trade image, volume, and scene files back and forth with them. Python version: Blender and the bpy module bind to a single Python version (so add-ons and VFX tools need only target one); Blender bundles its own complete Python and ignores the system Python by default, which --python-use-system-env can change if you set up a compatible version.

## Python Wheels

Python wheels (*.whl) are the standard way to distribute Python modules, and Blender supports them so extensions can be self-contained. Guideline: by convention, keep the files under ./wheels/. Requirements: bundle wheels unmodified from Python's package index; include their dependencies; match Python's binary-distribution filename specification (wheels downloaded from the package index already do); and use forward slashes as path separators when listing them in the manifest. How to bundle: download the wheel into ./wheels/ — for a platform-independent module, `pip wheel jsmin -w ./wheels`, and for one with compiled binaries, fetch a wheel for every supported platform, e.g.:
```text
pip download pillow --dest ./wheels --only-binary=:all: --python-version=3.13 --platform=macosx_11_0_arm64
pip download pillow --dest ./wheels --only-binary=:all: --python-version=3.13 --platform=manylinux_2_28_x86_64
pip download pillow --dest ./wheels --only-binary=:all: --python-version=3.13 --platform=win_amd64
```
(the platform identifiers are listed on pillow's download page). Then list the wheels as paths in blender_manifest.toml:
```text
wheels = [
"./wheels/pillow-12.1.0-cp313-cp313-macosx_11_0_arm64.whl",
"./wheels/pillow-12.1.0-cp313-cp313-manylinux_2_27_x86_64.manylinux_2_28_x86_64.whl",
"./wheels/pillow-12.1.0-cp313-cp313-win_amd64.whl",
]
```
Installing the package then extracts the wheel into the extension's own site-packages directory. To confirm it loads, import it in the Python console and print its location:
```text
import PIL
print(PIL.__file__)
```
Platform builds: wheels can bloat an extension, so you can build a separate zip per required platform using the build command's --split-platforms option, `blender --command extension build --split-platforms`. For example, a manifest excerpt
```text
id = "my_addon_with_wheels"
version = "1.0.0"

platforms = ["windows-x64", "macos-x64"]
wheels = [
"./wheels/pillow-12.1.0-cp313-cp313-macosx_11_0_arm64.whl",
"./wheels/pillow-12.1.0-cp313-cp313-manylinux_2_27_x86_64.manylinux_2_28_x86_64.whl",
"./wheels/pillow-12.1.0-cp313-cp313-win_amd64.whl",
]
```
yields my_addon_with_wheels-1.0.0-windows_x64.zip and my_addon_with_wheels-1.0.0-macos_x64.zip — and even though a Linux wheel is present, no Linux zip is built, because the platforms field lists only Mac and Windows.

## Extensions Tags

Different extension types have different available tags. The Add-on tags currently supported by the Blender Extensions Platform are: 3D View, Add Curve, Add Mesh, Animation, Bake, Camera, Compositing, Development, Game Engine, Geometry Nodes, Grease Pencil, Import-Export, Lighting, Material, Modeling, Mesh, Node, Object, Paint, Pipeline, Physics, Render, Rigging, Scene, Sculpt, Sequencer, System, Text Editor, Tracking, User Interface, and UV. The Theme tags are: Dark, Light, Colorful, Inspired By, Print, Accessibility, and High Contrast.

## Actions

Actions are Blender's container for animation data: when you animate, say, an object's location, that animation lives in an action rather than on the object itself, and the object uses the action to be animated — much as a mesh uses a material to be shaded. Every animatable data-block (objects, meshes, materials, and so on) works this way, storing no animation of its own but using an action that holds it. Actions are data-blocks too, so they can be appended or linked into other blend-files, which makes them useful not just for storage but for organizing and reusing animation — for instance, in a library of run cycles, jumps, and idles, each animation can be its own action to link or export separately. Action slots: an action's animation data is further organized into slots, with each action holding a set of slots and distinct data per slot; an animated data-block names both an action and a slot within it, and that pairing decides which animation data drives it. Slots let one action store separate data for several data-blocks — for example, a bouncing ball that changes color each bounce involves two data-blocks (the object and its material), and slots let both animations sit in one action, one slot each. You can have as many slots as you want for as many objects, materials, lights, and so on (a baked simulation of 100 bouncing balls could be one action with 100 slots), but you needn't use slots at all — 100 separate actions would work too. Still, animation data is always organized into slots, so an action needs at least one slot to animate anything. Slots aren't tied to a specific data-block: any data-block can use any slot, so two characters can share one slot in one action and both get driven by the same data — slots are simply a way to organize distinct animation data within an action, with no inherent attachment to anything in the scene. Internally an action's data is further split into layers and strips, which aren't currently in the UI and don't affect how you use actions today (it's groundwork for future animation features and safe to ignore), though they are exposed in the Python API, so script and add-on authors should be aware of them. Slot names and associated types: each slot has a name you can set freely (new slots default to the name of the last slot assigned to that data-block, or to the data-block itself if none has been). Each slot also has an associated data-block type it's meant for (material, object, etc.), set automatically when the slot is first assigned, and shown as an icon beside the slot's name in the action editor's channel list. Within an action a slot must have a unique name-plus-type combination, so two slots can both be named "Cube" only if one is for objects and the other for materials; if both are for objects, the matching type forces different names, and Blender uses the familiar "Cube" and "Cube.001" approach. It's possible, though awkward and not useful, to assign a slot to a data-block of the wrong type (say a material slot to an object) — nothing breaks, but that slot's F-Curves probably won't match the data-block's properties and so won't animate anything. F-Curves and channels: F-Curves are Blender's fundamental unit of animation and the main data an action holds, each containing keyframes that define how a property (like an object's X location) changes over time. Blender's animation editors (dopesheet, graph editor, and so on) show a channel list on their left of animated properties, and for actions those channels correspond to the F-Curves animating them.

Channels also support a limited grouping called "channel groups" — by default Blender groups each bone's channels, and while a few features rely on groups, they're mostly for convenience. Working with actions: when you first animate a data-block, Blender tries to find a suitable action (creating one if it can't), then makes and assigns a new slot for that data-block. Its heuristics assume closely related data-blocks usually share an action — for example, an object and its data are considered close, so if a camera object is already animated and you key its focal length (which lives on the camera data, not the object), the object's action is reused for the camera data, and vice versa. Other closely related pairs include materials together with their embedded node trees, worlds with theirs, and meshes alongside their shape-key data. The exception is a data-block with more than one user: if one mesh data-block is shared by several objects, the relationship is ignored and the mesh data and its users get separate actions despite otherwise counting as closely related. Manually assigning actions and slots: besides letting Blender choose, you can assign them yourself — selecting both action and slot to attach existing animation, or assigning the action but leaving the slot blank so a new slot is made when the first key is set. Each data-block in the properties editor has an Animation panel with action and slot selectors for this, and for the active object you can also assign them in the Action Editor's header (see also the Scene, Speaker, Movie Clip, Mask, and Node Group animation panels). When picking a slot you may not see every slot of an action, because Blender limits the dropdown to slots whose associated type matches the data-block. When you choose an action, Blender tries to auto-pick a fitting slot by name and type; if none fits, the selector stays empty and you can pick an existing slot, make a new one, or just start keying and let Blender create one — and if it assigns a slot you didn't want, pick another or clear the selection. Switching the action on multiple data-blocks: when an action animates several data-blocks you may need to swap it on all of them at once so animation that belongs together swaps together, for which there are two options — Replace Action (looks at the active object's action and swaps it on any object directly animated by it, excluding NLA strips and action constraints; found in the Action Editor) and Remap Users (in the outliner, right-click the action and pick Remap Users, which swaps every use — NLA strips and action constraints included — so apply it carefully). NLA: an action can also attach to NLA strips inside a data-block's NLA system; see the NLA Editor docs for animating via NLA. Action properties: you can manually set an action's intended useful frame range from a panel in the Dope Sheet or NLA Editor while a channel or an NLA track is active. Manual Frame Range specifies the intended playback range (used by some tools but not affecting evaluation), toggled by a checkbox; when set, it's used instead of the actual keyframed range when adding an NLA track based on the action, and exporters can use it to choose which frames to export. The range appears in the editor background as a diagonal hash pattern, setting it apart from the solid fill used for the current playback range, and its values are usually integers but may be fractional. Cyclic Animation marks the action as intended to loop over the range, where the first and last frames should be the same pose one loop apart (so the range includes the duplicated initial key); this signals intent and doesn't itself make the action cycle, but with Cycle-Aware Keying enabled it automatically turns on cyclic extrapolation and sets the loop period for newly added curves.

Action slot properties (for the slot used by the currently selected channel-list item): Name (the slot's name) and Type (the data-block type the slot is meant to animate). Custom properties: define and manage your own properties to hold data on the action's data-block; the Custom Properties page has more.
