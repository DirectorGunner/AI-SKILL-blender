# Uv Layout, Gltf 2 0, Add Ons

> Blender Material Nodes and Images reference. Original prose; identifiers preserved verbatim.

## Overview

Covers: UV Layout; glTF 2.0; Add-ons.

## UV Layout

The UV Layout add-on saves a mesh's UV map out as an image through UV Editor ▸ UV ▸ Export UV Layout. It is on by default (otherwise enable "UV Layout" under Preferences ▸ Add-ons) and can write PNG, EPS, or SVG. Pick the faces in the 3D Viewport (not in the UV Editor). The drawing it produces is just the UV edge lines that land inside the UV Editor's standard grid; anything past that border is skipped, selected or not. People typically open that image in a paint program as a see-through guide while painting a texture, then load the finished texture back as part of a Blender material (see Image Textures for using images as textures).

Its properties: "All UVs" writes every UV instead of just the 3D-Viewport selection; "Export Tiles" chooses between the single [0,1] range and the complete set of UV tiles; "Modified" takes the UVs after the mesh's modifiers have been applied; Format is the file type (.png/.eps/.svg); Size is the pixel resolution; and Fill Opacity controls how opaque the fill is.

## glTF 2.0

The glTF 2.0 add-on (on by default) handles .glb and .gltf through File ▸ Import/Export ▸ glTF 2.0 (.glb, .gltf). glTF ("GL Transmission Format") exists to ship and load 3D models in web and native apps with little decode work, trimming both size and load cost; it's widespread on the web and read by engines like Unreal Engine 4, Godot, and Unity3D. What the add-on covers: meshes; Principled-BSDF and Shadeless (Unlit) materials; textures; cameras; the punctual light types (point, spot, directional); the extensions listed below; extras (custom properties); and animation across keyframes, shape keys, and skinning.

Meshes. Since glTF's layout echoes the buffers a GPU uses for real-time drawing, export triangulates quads and n-gons automatically; vertex counts can creep up versus Blender wherever UVs break or edges are flat-shaded (those vertices get split); and curves and other non-mesh data don't survive — convert them to meshes beforehand. GPU Instances ride the EXT_mesh_gpu_instancing extension when enabled, within limits: an instance has to be a mesh with no children of its own, every instance must parent to the same object, and instances are recognized by shared mesh data (the extension can't vary materials, so all may share one set). On import they're rebuilt by re-sharing mesh data.

Materials. glTF's core is a metal/rough PBR set of channels: Base Color, Metallic, Roughness, baked Ambient Occlusion, a tangent-space Normal Map (+Y up), and Emissive; extras beyond that ride extensions. Imported materials differ from Blender's, so the add-on assembles a node network that approximates each one as closely as it can — it understands core Metal/Rough, Spec/Gloss (KHR_materials_pbrSpecularGlossiness), and a few extension materials, and studying an import is a good way to learn the node setups that export cleanly. Exports support core Metal/Rough and Shadeless (KHR_materials_unlit), built from whatever nodes the exporter recognizes. Base Color is pulled from the Principled BSDF's Base Color (its default swatch if nothing is wired); a connected Image Texture is used when present; failing that it can come from the Principled default, an RGB node, or an Ambient Occlusion node feeding Base Color (glTF ignores the AO socket, but its color output works as a base color). Metallic and Roughness come off the Principled node (0.0–1.0 sliders when unwired); supplied as an image, glTF wants metallic on blue and roughness on green within one image, so the suggested wiring runs a Separate RGB node with green into Roughness and blue into Metallic and the texture set to Non-Color — the exporter spots this as standard and copies the image straight through. For baked occlusion: glTF can store an AO map, yet no node graph makes Blender consume it exactly as glTF means it; if a custom group named "glTF Material Output" carries an Occlusion input, an Image Texture there becomes the occlusion map (you needn't see the effect in Blender).

glTF export channels, continued. A pre-baked occlusion map feeds a node that never renders but still exports; enabling Shader Editor Add-ons in preferences lets you add the "glTF Material Output" group via Add ▸ Output ▸ glTF Material Output. Occlusion lands on glTF's red channel and may ride one image together with roughness and metallic, echoing how glTF packs the three; Cycles' Bake panel can produce that AO map. For Normal Map: route an Image Texture's color into a Normal Map node, then its normal output into the Principled normal socket, with the texture on Non-Color. Leave the Normal Map on its default Tangent Space (glTF supports nothing else); its strength is adjustable, and the exporter doesn't ship these nodes — it just reads them to find the image and copy the strength. Cycles' Bake panel (Bake type Normal, default spaces) can bake tangent-space normals for this. Emissive: wire an Image Texture into the Principled Emission, or into an Emission shader optionally combined with the Principled output through an Add Shader; when emission stands alone, drop Base Color to black and Roughness to 1.0 so other channels don't intrude; any emissiveFactor component above 1.0 pulls in KHR_materials_emissive_strength. Clearcoat: a nonzero default or connected texture on Clearcoat brings KHR_materials_clearcoat (also reading Clearcoat Roughness); image data wants clearcoat on red and its roughness on green, monochrome images are remapped, and the Clearcoat Normal takes the same +Y tangent-space map (reuse the base one, give it its own, or leave it off for a smooth coat), with every clearcoat texture on Non-Color. Sheen: a Sheen Roughness texture must carry values on alpha (the Sheen BSDF is Cycles-only, so you may switch to Cycles to add it). Specular: a non-default Specular IOR Level or Specular Tint — value or texture — triggers KHR_materials_specular. Anisotropy is converted both ways (helper nodes appear on import and are detected on export); grayscale textures can drive the Anisotropic and Anisotropic Rotation sockets for conversion, and the tangent socket has to link to a tangent node using the normal map's UV map. Transmission: a nonzero or textured Transmission input brings KHR_materials_transmission (a texture stores values on red, Non-Color); unlike alpha blending it keeps specular reflections at full strength. Within glTF, alpha blending represents geometry that's physically absent (gauze, say) while transmission stands for solid stock that passes non-specular light (glass); base roughness can frost it; a transmissive material's blend mode normally stays Opaque unless it only partly covers the surface; and real-time engines impose limits on what shows through (one transmissive surface may hide another). Textures on these channels should sit on Non-Color.

More on glTF materials. Refraction needs KHR_materials_transmission paired with KHR_materials_volume (see the Volume docs). Transmission is hard for real-time engines, and KHR_materials_transmission isn't broadly supported yet. IOR: import sets the Principled IOR to glTF's default 1.5 when KHR_materials_ior is absent, or to the stored value when present; export writes IOR only alongside transmission, volume, or specular, and omits 1.5 as the default. Volume: drive it with a Volume Absorption node on the Output's Volume socket (data via KHR_materials_volume) — the Principled node needs some transmission first; the absorption color becomes glTF's attenuation color (no texture permitted), its density becomes the reciprocal of attenuation distance, thickness plugs into the glTF Material Output group's Thickness socket, and a thickness texture must use green. glTF Variants (switch the UI on in add-on preferences) live in two spots — the 3D View's glTF Variants tab and, for advanced bits, Mesh Material Properties — and the central idea is that each material slot equals one glTF primitive. After importing a file carrying KHR_materials_variants you can list every variant, jump to one with Display Variant, or return to the no-variant materials with Reset to default. Add a variant from the right of the list and rename it with a double-click; once you've swapped materials in the slots, "Assign to Variant" pins the current set to the active variant, while "Assign as Original" marks the defaults (exported as the fallback that variant-unaware viewers show). The advanced tab reports, per slot, which variants reference the material and lets you bind materials to variants, though the 3D View tab is the recommended place. Double-sidedness: for front-faces-only materials, switch on Backface Culling (Camera) in an EEVEE material's Settings panel (with Cycles or Workbench, hop to EEVEE to set it and back); leaving it off keeps the material double-sided, because glTF's doubleSided flag is the inverse of this toggle. Alpha modes — three of them, picked automatically from what's on the Alpha socket. Opaque keeps alpha at 1. Mask forces alpha to 0 or 1 for hard cutouts (leaves, holey cloth): round alpha with a Math node (0.5 and up rounds to 1), and a different threshold is possible through more Math nodes; it amounts to EEVEE's Alpha Clip but built from shader nodes so other engines honor it. Blend gives graded transparency that tints what's behind (think colored film), but it's costly to render and viewers may glitch on busy scenes — so quarantine the parts that can be Opaque or Mask and reserve Blend for what truly needs it, or keep one transparent layer ahead of opaque geometry. UV mapping: a UV Map node plus a Mapping node on an Image Texture gives control; Mapping settings export through KHR_texture_transform, where Point is the advised type (Texture and Vector also work), the offsets that carry are Location X/Y, Rotation on Z only, and Scale X/Y, and the Texture type demands equal X and Y scale. These nodes are optional, and not every reader handles several UV maps or texture transforms.

glTF export, odds and ends. Factors: an Image Texture can be scaled by a constant color or scalar, written into the glTF as factors (numbers multiplied with the texture) — uncommon; for a scalar use a Math node set to multiply with the second value as the factor, and for a color use a Mix node (color/multiply) with Factor at 1 and Color2 (B) as the factor. One material can combine all of the preceding at once. UDIM keeps several textures in one file; Blender supports it but glTF doesn't, so export splits the image per tile and rewires the nodes to suit — all UDIM textures must share a UV map. Shadeless (Unlit): to get one, blend in a camera ray and steer clear of the Principled BSDF (it exports as KHR_materials_unlit and reads flat in Blender). Lights: three punctual kinds export — point, spot, and directional (Blender's Sun); Area lights and World lighting aren't supported and drop out. Tick Include Punctual Lights to send them (via KHR_lights_punctual). A glTF light's color multiplies the light object's color by an Emission shader's color output where the light is node-driven; under EEVEE only the object's color field counts and the Emission node is ignored (to mirror the viewport), whereas through Blender 5.1 the node graph outranks the color field (so you might delete it to match the viewport). Intensity for Sun lights comes from the object's strength field; point and spot lights can also read a Light Falloff node's Strength; under EEVEE the node graph is ignored and only the strength field counts. Extensions: glTF's core can be widened with extensions to hold details that weren't universal at launch, and not every reader supports every one; some Blender features export only through them, and the add-on speaks a range on import and/or export. A Python developer can teach Blender new glTF extensions in a separate add-on without editing this one (there's a GitHub example, and you can register an extension prefix if needed). Custom Properties: always imported, and exported off most objects when Include ▸ Custom Properties is on, stored in the glTF object's extras; unlike extensions, extras have no fixed namespace and serve any purpose. Animations: a glTF animation drives object or pose-bone transforms or shape-key values, one animation may touch several objects, and a file may hold many. On import the first plays straight away (scrub the Timeline); the rest organize into the Nonlinear Animation editor, each becoming an action stashed on an NLA track named for the glTF animation and made audible with Solo (the star). When one animation spans objects it's split into parts, each stashed on its object under the shared track name; Solo all those tracks to play the whole. Each animation imports as one action, with several slots when it spans objects (a slot for TRS, another for shape keys, and so on); there's currently no way to see a model's neutral, non-animated pose once it has carried animation. A Dope Sheet switcher (enable its UI in preferences) flips between all imported animations, turning Solo on for the tracks involved and resetting unanimated objects to Rest.

glTF animation export hinges on the Animation ▸ Mode option, which decides how clips are derived from actions and the NLA. Actions (the default): an action exports when it's the object's active action or has been stashed to an NLA track (Stash or Push Down in the Action Editor); anything not bound to an object either way is skipped, so stash whatever you need. A clip's name defaults to the action name but can be overridden by renaming the NLA track; give two tracks on two objects the same name and they fold into one clip that runs together. The importer arranges actions to round-trip correctly in this mode, which fits exporting a character's clip library for a game engine, one action per track. Up to Blender 4.4 tracks merged by name; from 4.4 onward, with slotted actions, the default merges by the action used rather than by name. Active Actions Merged: ignore the NLA arrangement and export a single clip built from every object's active action. NLA Tracks: each track exports as its own clip — useful with strip modifiers or several actions on one track — and again, identically named tracks across objects merge and run together. Scene: clips export exactly as the viewport plays them, either as one clip or per object. Only some animation carries over — object transforms (location, rotation, scale), pose bones, and shape-key values — so physics, lights, and material animation are dropped. Sampling a shape-key animation that's driven by bone transforms requires the mesh to be a direct child of the bones' armature, and only the Actions and Active Actions Merged modes cope with unsampled animation.

File-format variants. The importer reads them all; the exporter makes you choose one. glTF Binary (.glb): everything — mesh, textures, the rest — packed into a single binary, which is the easiest to copy or share. glTF Separate (.gltf + .bin + textures): a JSON .gltf for structure, a .bin for mesh and vector data, and optional .png/.jpg textures the .gltf points at; this is the friendliest for later hand-editing of the JSON or images, but the pieces have to travel together. glTF Embedded (.gltf): one JSON file with mesh and image data Base64'd inside, useful over plain-text-only links; it's the bulkiest option and should be reserved for when it's required, and it only appears once you enable it in preferences.

Import properties. Merge Vertices: glTF has to split vertices wherever normals, UVs, or other attributes are discontinuous (graphics hardware needs that), and this tries to rejoin co-located ones, short of merging vertices whose normals differ. Shading picks how normals are derived on import. Lighting Mode offers fallback behavior for non-standard engines (it affects lights): Standard (physical glTF units — cd, lx, nt), Unitless (non-physical, for when exposure controls aren't around), or Raw (deprecated; Blender's strengths with no conversion).

Further glTF settings. Import, Texture: Pack Images folds all images into the blend-file; "Import WebP textures" loads a WebP in place of the PNG/JPG fallback when one exists; "Import unused Materials & Textures" brings in everything, used or not. Import, Bones & Skin: Bone Direction changes the tip-placement heuristic (Fortune can misbehave on non-uniformly scaled models and is otherwise cosmetic; the default leaves axes alone and re-exports cleanly, adjusting display for clarity even when the original axes are less exact); "Guess Original Bind Pose" works out the Edit-Mode pose of bones (and thus skinned meshes) by inferring the pose behind the inverse bind matrices; "Disable Bone Shape" hides bone shapes in the 3D View; "Bone Shape Scale" resizes them. Import, Pipeline: "Import Scenes as Collections" (the default) maps glTF scenes to collections — one scene drops objects into the active collection; several scenes each become a collection (non-default ones excluded from the View Layer), and stray nodes belonging to no scene gather in an Orphan Collection (also excluded); switched off, the glTF scene loads into the active scene and any others arrive as new Blender scenes. Two more: "Select Imported Objects" leaves the new objects selected, and "Import Scene Extras" turns glTF extras into scene-level custom properties.

Export. Format chooses the file variant (see above). "Keep Original" (Separate only) reuses existing texture files where it can (caveat: where the PBR standard expects one texture but you've used several, only one is taken, which may surprise you). Textures (Separate only) is the texture folder, relative to the gltf. Copyright stores the model's legal terms. "Remember Export Settings" saves them into the blend-file for next time. Include filters the export to Selected, Visible, or Renderable objects, the Active Collection (with Include Nested Collections to recurse), or the Active Scene, plus Custom Properties (as extras), Cameras, and Punctual Lights (directional/point/spot via KHR_lights_punctual). Transform offers Y Up to export in glTF's +Y-up convention. Data – Scene Graph: Geometry Nodes Instances (experimental), GPU Instances (EXT_mesh_gpu_instancing), Flatten Object Hierarchy (for TRS matrices that won't decompose; only skinned meshes stay parented to the armature), and Full Collection Hierarchy (collections export as empties keeping the whole tree; an object in two collections is written once, in the first). Data – Mesh: Apply Modifiers (the evaluated mesh), UVs, Normals, Tangents, Attributes (names beginning with an underscore), Loose Edges (as lines on the first material slot), Loose Points (as glTF points on the first slot), and Shared Accessor (a shared triangle index accessor, leaner when materials are many). Data – Mesh – Vertex Color: Use Vertex Color can be Material (export the color used as a Base Color multiplier in the tree — default, and most spec-faithful), Active (export the active colors even if unused; a compliant viewer treats them as a Base Color multiplier), Name (export a named set), or None; "Export all vertex colors" adds the extras as `COLOR_1`, `COLOR_2`, and so on, and there's an option to export the active color when no material is present. Data – Material: Materials can be Export (the full thing — every texture and shader from the tree), Placeholder (just the placeholder, no textures or shaders, with primitives kept apart so slot info survives), Viewport (only Base Color, Roughness, Metalness), or No Export (primitives merged, slot info lost); Images sets the output format (PNG is lossless and usually preferred, JPEG suits smaller web files; choosing WebP makes every texture WebP with no PNG/JPG fallback; None exports materials with no textures); Image Quality covers JPEG/WebP; Create WebP adds WebP versions on top of the existing textures (already-WebP ones are untouched); WebP fallback adds a PNG fallback for each WebP; "Unused images" exports images no material uses; "Unused textures" exports texture info (sampler, image, texcoord) no material uses. Data – Shape Keys writes the morph targets, with Shape Key Normals and Shape Key Tangents; the Optimize options are "Use Sparse Accessor if better" (when it shrinks the file) and "Omitting Sparse Accessor if data is empty" (off by default, since some viewers mishandle it). Data – Armature: "Use Rest Position Armature" exports with the rest pose as the joint rest (off uses the current frame's pose), "Export Deformation Bones only" (their animation gets baked), "Remove Armature Object" (where feasible — not when an armature has several root bones), and "Flatten Bone Hierarchy" (for TRS matrices that won't decompose). Data – Skinning: export skinning data; Bone influences sets how many joint influences go out (other than 4 or 8, many viewers render it wrong); "Include All Bone Influences" exports them all (again, may render wrong). Data – Lighting: Lighting Mode (Standard physical units, Unitless, or deprecated Raw). Data – Compression: Google Draco mesh compression, with a Compression Level (higher is slower both ways) and per-attribute quantization for Position, Normal, Texture Coordinates, and Color (higher gives better ratios).

The last glTF export controls (note: across these, higher quantization values compress better). Animation: Animation mode selects the export mode (per the Animations section). Bake & Merge: "Bake All Objects Animations" helps where objects are constrained yet not animated in their own right; Merge Animation combines clips by Action (through slots), by NLA Track Name, or not at all — by track name, same-named animations join; by action, same-action ones join; with no merge, each exports on its own. Rest & Ranges: "Use Current Frame as Object Rest Transformations" exports at the current frame (off uses frame 0 as the object rest); "Limit to Playback Range" trims clips to the chosen playback range; "Set all glTF Animation starting at 0" (good for loops); and Negative Frames either slides or crops anything sitting below frame 0. Armature: "Export all Armature Actions" sends every action bound to one armature (it can't handle exports with more than one armature); "Reset pose bones between actions" clears pose bones between exported actions, which you need when some bones aren't keyed in every clip. Shape Keys: "Shape Keys Animations" exports shape-key animation (the shape keys themselves must be exported), with a matching "Reset Shape Keys between actions" for clips that don't key every shape key. Sampling: it applies to all clips (skip it and the export can come out wrong); Sampling Rate is how often (in frames) animated values are read; Sampling Interpolation Fallback sets the interpolation for unkeyed properties (LINEAR or STEP/CONSTANT). Optimize: "Optimize Animation Size" shrinks the file by dropping keyframes duplicated across a channel; "Force keeping channel for armature / bones" preserves a minimal track when a rig's keyframes are all the same; "Force keeping channel for objects" does the same for object transforms; "Disable viewport for other objects" turns their viewport off during export for speed where it can. A Filter restricts which actions export by name. Collection Exporters: this exporter doubles as a collection exporter (see Collections) — the Include block doesn't apply (as with the others), there's an option to export about the collection's center (the center of mass of its root objects), and the collection's custom properties export as scene glTF extras. The importer/exporter itself is built in the glTF-Blender-IO project (file bugs, request features, or contribute there), while the format's own design is discussed and developed at the Khronos Group's glTF repository.
